# Архитектура конвейера редактуры

> **Ознакомительный документ для подключения LLM к многоагентной системе pipeline.**
>
> Этот документ описывает структуру работы многоагентного конвейера для редактуры научных текстов и статей. Внимательно изучи его — после ознакомления ожидай конкретных инструкций по своей роли.
>
> Конвейер **не привязан к конкретным моделям**. Любая LLM-модель или человек может быть назначена на любую роль — назначение происходит при начале конкретного диалога.

## 1. Роли в конвейере

Конвейер построен по модели **split-and-merge** с распределением задач по 4 функциональным ролям.

### Роль 1: Оркестратор

**Назначение:** пользователь (человек) или LLM-помощник.

**Функции:**

- Декомпозиция задачи на подзадачи
- Маршрутизация подзадач по исполнителям (текстовик vs. механика)
- Принятие стратегических решений (что править, что оставить, какую модель/инструмент подключать следующей)
- Контроль результатов после каждой итерации
- Решение о переходе к следующему этапу или возврате на доработку
- Определение роли для каждой подключаемой модели в каждом конкретном сеансе

**Принципы работы оркестратора:**

- Не выполняет содержательную работу сам — только координирует
- Имеет право вето на любое решение со-оркестратора и рецензента
- Может снять любой пункт из чек-листа правок (если пользователь явно согласен)
- Выбирает инструмент под задачу, а не наоборот

---

### Роль 2: Рецензент / Валидатор правок

**Назначение:** любая LLM-модель с доступом к файловой системе и способностью запускать скрипты (например, через `python-docx`).

**Функции:**

- Критический разбор исходного документа → список нарушений с группировкой по критичности (критические / серьёзные / прочие)
- Программная вычитка промежуточных результатов через `python-docx`:
  - проверка полей страницы (EMU → мм)
  - проверка шрифтов и кеглей по всем абзацам и ячейкам таблиц
  - проверка междустрочных интервалов
  - проверка наличия кириллицы вне `[квадратных скобок]` в References
  - подсчёт слов в аннотациях
  - проверка нумерации ссылок в таблице vs. в тексте
- Оценка выходов текстовика по шкале 0–10 с детализацией по блокам
- Подготовка точечных промптов-исправлений для технического исполнителя
- Финальная проверка по чек-листу из N пунктов

**Методология оценки:**

- Сравнение по 15+ критериям в единой таблице
- Выявление эмпирических закономерностей (например: «большая модель не обязательно лучше для задач с жёсткими требованиями»)
- Каждый недостаток сопровождается конкретным примером из вывода модели и предлагаемым исправлением

---

### Роль 3: Текстовик

**Назначение:** любая LLM-модель, способная генерировать длинные тексты с соблюдением формальных ограничений (лимиты слов, транслитерация по ГОСТ, стилистика).

**Функции:**

- Написание N текстовых фрагментов по чёткому ТЗ от оркестратора
- Каждый фрагмент промаркирован `[TEXT-N]` для последующей вставки
- Фрагменты **не вставляются в документ** — только генерируются как текст с метками

**Требования к качеству:**

- Строгое следование лимитам (например, 95–105 слов в аннотации)
- Фактологический стиль аннотаций: что сделано, на каком материале, какие результаты, какие выводы
- Корректная транслитерация по ГОСТ 7.79–2000 (если применимо):
  - `ь` → `'` (апостроф)
  - `ц` → `ts`
  - `ч` → `ch`
  - `ш` → `sh`
  - `щ` → `shch`
  - `х` → `kh`
  - `й` → `y` (или `j` — последовательно)
- Введение аббревиатур при первом упоминании: `(ИИ)`, `(AI)`, `(ЭМК)`, `(EHR)` и т. п.
- Числа ≥ 10 — цифрами, не прописью
- Все диапазоны (лет, страниц) — через en-dash `–`, не дефис `-`
- Без точек в концах списков ключевых слов и названий таблиц

**Типичные антипаттерны (что нужно ловить рецензенту):**

- Конструкции «в статье представлен», «this review analyzes», «требует тщательного осмысления» — **запрещены** в аннотациях
- Замена авторов и заголовков в библиографии на «более подходящие» (галлюцинация)
- Пропуск апострофов в транслитерации мягкого знака
- Числа прописью вместо цифр
- Точки в концах списков ключевых слов и названий таблиц
- Недобор слов в аннотациях
- Философский стиль вместо фактологического
- Самовольное добавление фактов, которых нет в исходнике

---

### Роль 4: Технический исполнитель

**Назначение:** любая LLM-модель или инструмент, способные работать с `.docx`-файлами (через `python-docx`, OOXML-редактирование, специализированные CLI-утилиты типа OfficeCLI и т. п.).

**Функции:**

- Механические операции над документом по готовому чек-листу
- **Не придумывает текст** — берёт дословно из `[TEXT-N]` блоков, подготовленных текстовиком
- Сохраняет результат как `.docx`

**Типичные операции:**

- Глобальное форматирование (поля, шрифты, интервалы, отступы)
- Добавление заголовков разделов
- Замена inline-подзаголовков на полноценные заголовки
- Вставка готовых текстовых фрагментов в нужные места
- Простановка нумерации в списках литературы
- Расшифровка аббревиатур
- Вставка англоязычного блока в конец документа

**Известные риски (что нужно перепроверять рецензенту):**

- Операции «замени X на Y в конкретной ячейке таблицы» — иногда игнорируются
- Операции «добавь фрагмент И одновременно удали старый» — могут приводить к дублированию
- Потеря данных в таблицах при использовании сложных pack/unpack-инструментов
- Самоотчёт модели не всегда соответствует факту — обязательна программная проверка

### Роль 5: Факт-чекер (обязательный гейт + вызываемый инструмент)

**Назначение:** фиксированная связка **FreeBuff + OpenAlex + Crawl4AI + Playwright MCP + Scrapling + researcher-web** (не назначается на другую модель).

**Тип:** двойственный. По умолчанию — **обязательный гейт** на Этапе 4.5 (для любых задач, содержащих библиографию или веб-ссылки). По требованию Оркестратора + Рецензента — **вызываемый инструмент** для точечной проверки в любой точке конвейера (например, при подозрении на галлюцинацию в уже готовом документе).

**Функции:**

- 100% проверка фактов и текстов `[TEXT-N]` **до** их вставки в `.docx`
- Сверка библиографии (авторы, заголовки, годы, DOI) с исходником
- Подтверждение написания брендов (например, `CELSUS` на celsus.ai), официальных английских названий журналов, адресов организаций
- Прямой парсинг целевых страниц через Crawl4AI + Playwright MCP + кросс-проверка через `researcher-web`. **DOI сначала проверяются через OpenAlex API (Ур.0.5)** — мгновенно, без парсинга. **Интерактивные страницы (логин, формы) — через Playwright MCP (Ур.2.5).** **Cloudflare/anti-bot — через Scrapling (Ур.3).**
- Возврат вердикта по каждому факту: `CONFIRMED` / `REFUTED` / `UNCERTAIN` с прямой цитатой и URL источника

**Принципиальные ограничения:**

- НИКОГДА не выдумывает подтверждения. Если не нашёл — пишет `UNCERTAIN`
- НЕ утверждает, что факт галлюцинация, если не проверил прямым URL. Пишет: «Требуется ручная проверка»
- Если Crawl4AI вернул маркер блокировки (`ip заблокирован`, `cloudflare`, `security verification`) — передаёт URL на Scrapling (Ур.3). Если контент скрыт за формой/логином — использует Playwright MCP (Ур.2.5). Если обе альтернативы не сработали — использует альтернативные источники (`cyberleninka.ru`, официальный сайт журнала) и помечает результат как `BLOCKED`, а не `CONFIRMED`

> **Полная спецификация роли, схема вызова и методология работы — в [разделе 6](#6-факт-чекер--обязательный-гейт).**

### Вспомогательные суб-агенты (в окружении FreeBuff)

В окружении FreeBuff суб-агенты работают **всегда** — FreeBuff сам автоматически вызывает их по ходу задачи (например, `basher` для команд, `code-reviewer` после правок кода, `thinker` для нетривиальных решений). Оркестратору не нужно явно вызывать каждый суб-агент.

Однако **последовательность вызова конкретных суб-агентов можно и нужно явно прописывать в промпте**, чтобы направлять работу FreeBuff и избегать катастроф:

- **Промпт v2 (катастрофа):** без явных инструкций о том, когда и как использовать `researcher-web` → агент вышел из-под контроля и фальсифицировал 9 из 27 источников, заменяя авторов на «более подходящие»
- **Промпт v3 (успех):** с явными запретами («НЕ ИСКАТЬ статьи, НЕ ИСКАТЬ авторов, НЕ ЗАМЕНЯТЬ источники») и встроенной библиографией → фальсификация устранена, 8/10 качество

| Суб-агент | Назначение | Риск при свободном вызове |
|-----------|------------|---------------------------|
| `basher` | Запуск shell-команд, Python-скриптов | Низкий — детерминированный исполнитель |
| `researcher-web` | Проверка фактов в интернете (названия журналов, брендов, адреса) | **Высокий** — может начать искать «более подходящие» источники и фальсифицировать данные. Требует явных запретов в промпте |
| `researcher-docs` | Чтение технической документации (ГОСТ-стандарты, спецификации) | Средний — может выйти за рамки задачи |
| `code-reviewer` | Проверка качества сгенерированных блоков перед выводом | Низкий — но может зациклиться на мелочах |
| `thinker` | Глубокое рассуждение для нетривиальных решений | Средний — может переусложнить простую задачу |

> **Важно:** суб-агенты — это не «вызываемые инструменты по требованию», а **постоянная часть окружения FreeBuff**. Оркестратор не управляет каждым вызовом, но **формирует промпт так, чтобы направить суб-агентов в нужное русло**. Плохой промпт → катастрофа (v2). Хороший промпт с явными ограничениями → успех (v3).

## 2. Твоя роль в этой структуре

> Ты в этой структуре — **[РОЛЬ БУДЕТ УКАЗАНА ПРИ ОТПРАВКЕ]**

**Возможные роли:**

- **ТЕКСТОВИК** — ожидаю от тебя генерации текстовых фрагментов `[TEXT-N]` по чёткому ТЗ. Не вставляй фрагменты в документ — только генерируй как текст с метками.
- **РЕЦЕНЗЕНТ** — ожидаю от тебя критического разбора документа или оценки выходов другой модели по шкале 0–10 с детализацией.
- **СО-ОРКЕСТРАТОР** — ожидаю от тебя рекомендаций по разделению задачи и подготовке промптов для других ролей.
- **ТЕХНИЧЕСКИЙ ИСПОЛНИТЕЛЬ** — ожидаю от тебя механических правок в `.docx` по готовому чек-листу. Не придумывай текст — бери дословно из `[TEXT-N]` блоков.
- **ФАКТ-ЧЕКЕР** — вызывается точечно, не является постоянным участником цикла. Исполняется только связкой FreeBuff + OpenAlex + Crawl4AI + Playwright MCP + Scrapling + researcher-web. См. [раздел 6](#6-факт-чекер--обязательный-гейт).

Конкретная роль, задача и дополнительные уточнения будут указаны в следующем сообщении после этого ознакомительного блока.

## 3. Стандартная последовательность редактуры

Конвейер применяется для любого документа/текста, требующего приведения к формальным требованиям (ГОСТ, журнальный стиль, внутренние стандарты организации). Не обязательно связан с конкретным проектом — последовательность универсальна.

### Этап 1. Критический разбор исходника

**Исполнитель:** Рецензент

**Действия:**

- Чтение исходного документа (`.docx`, `.pdf`, `.md` — любой формат)
- Сравнение с требованиями (`task.txt`, ГОСТ, руководство для авторов)
- Формирование списка нарушений с группировкой:
  - **Критические** — структурные (отсутствие обязательных разделов, несоответствие шаблону, фальсификация данных)
  - **Серьёзные** — оформление (шрифты, поля, интервалы, нумерация)
  - **Прочие** — стилистические, мелкие отступления

**Результат:** структурированный список нарушений с конкретными примерами из документа.

---

### Этап 2. Декомпозиция задачи

**Исполнитель:** Оркестратор + Рецензент

**Действия:**

- Разделение списка нарушений на две категории:
  - **«Творческие»** — требуют написания нового текста (аннотации, ключевые слова, переводы, транслитерация, новые разделы)
  - **«Механические»** — требуют операций над документом (форматирование, заголовки, нумерация, вставка)
- Подготовка промпта-1 для текстовика с описанием каждого фрагмента и его требований
- Подготовка промпта-2 для технического исполнителя с заглушками `[TEXT-N]` в местах, где нужны текстовые фрагменты

**Принцип:** подбирать модель по сильной стороне. Не пытаться сделать всё одной моделью.

---

### Этап 3. Сбор фактологической базы (Pre-generation Fact-Checking)

**Исполнитель:** Факт-чекер (FreeBuff + OpenAlex + Crawl4AI + Playwright MCP + Scrapling + researcher-web)

**Тип:** обязательный гейт для задач, содержащих библиографию или веб-ссылки.

**Действия:**

- Извлечение всех упомянутых в исходном документе авторов, статей, DOI, URL и брендов
- **Программная проверка DOI через OpenAlex API (Ур.0.5)** — мгновенный lookup метаданных без парсинга страниц
- Прямой парсинг целевых страниц (Crawl4AI) и поиск (researcher-web) для сбора **сырого контента** (тексты статей, метаданные, подтверждающие существование источников)
- Формирование файла `results.md` с комплексом проверенных ссылок и материалов (сырой контент)

**Результат:** `results.md` — 100% валидированная фактологическая база (сырой контент), передаваемая Текстовику.

---

### Этап 4. Генерация текстовых фрагментов на основе валидированных данных

**Исполнитель:** Текстовик

**Действия:**

- Получение промпта-1 и файла `results.md` (сырой контент) от Факт-чекера
- Генерация фрагментов с метками `[TEXT-1]`, `[TEXT-2]`, ..., `[TEXT-N]` **строго на основе** предоставленных проверенных материалов (НИКАКИХ выдуманных ссылок или DOI)
- Строгий формат вывода: только промаркированные блоки, без комментариев

**Результат:** текстовый вывод с N блоками `[TEXT-N]` (собранными из валидированного сырья).

---

### Этап 4.5. Финальная верификация сгенерированных фрагментов (Post-generation Fact-Checking)

**Исполнитель:** Факт-чекер (FreeBuff + OpenAlex + Crawl4AI + Playwright MCP + Scrapling + researcher-web)

**Тип:** обязательный гейт.

**Когда вызывать:**

- После генерации `[TEXT-N]` Текстовиком (особенно `[TEXT-8]` REFERENCES)
- При подозрении на галлюцинацию (несоответствие авторов, годов, названий)
- Перед финальной сборкой `.docx`

**Действия:**

- Сверка авторов, заголовков, годов, DOI в сгенерированных `[TEXT-N]` с исходной фактологической базой (`results.md` из Этапа 3)
- Проверка написания брендов латиницей (через прямой парсинг Crawl4AI)
- Подтверждение официальных английских названий журналов
- Проверка существования статей (отсутствие ложных отрицаний: «может быть галлюцинацией» запрещено без проверки прямым URL)

**Результат:** отчёт `results_final.md` с вердиктом по каждому факту: `CONFIRMED` / `REFUTED` / `UNCERTAIN` / `BLOCKED` + прямые цитаты и URL источников. Оркестратор + Рецензент принимают решение: продолжить или вернуть текстовику на доработку.

---

### Этап 5. Рецензирование фрагментов

**Исполнитель:** Рецензент

**Действия:**

- Оценка каждого блока `[TEXT-N]` по шкале 0–10
- Проверка:
  - соответствия формальным требованиям (длина, стиль, формат)
  - соответствия исходному документу (особенно для библиографии — авторы, заголовки, годы **не должны** заменяться)
  - транслитерации по ГОСТ 7.79–2000 (если применимо)
  - введения аббревиатур при первом упоминании
  - отсутствия типичных антипаттернов («в статье представлен», «this review analyzes», числа прописью, точки в концах списков)
- Подготовка точечных правок и/или создание дополнительных блоков (как `[TEXT-9]` для INFORMATION ABOUT THE AUTHORS)

**Результат:** либо подтверждение качества (≥ 7/10), либо список правок + исправленные версии проблемных блоков.

---

### Этап 6. Сборка итогового промпта для исполнителя

**Исполнитель:** Оркестратор + Рецензент

**Действия:**

- Подстановка исправленных `[TEXT-1]` ... `[TEXT-N]` в промпт-2 вместо заглушек
- Формирование итогового промпта-2 с конкретными текстами для вставки

---

### Этап 7. Механическая вставка в документ (1-й прогон)

**Исполнитель:** Технический исполнитель

**Действия:**

- Получение итогового промпта-2 + исходного документа
- Выполнение N правок по готовому чек-листу
- Сохранение результата как `.docx`

**Ожидаемый результат:** 80–90% правок применено корректно.

**Типичные непройденные пункты:** замены в ячейках таблиц, операции «добавить + удалить», выравнивание отдельных заголовков.

---

### Этап 8. Программная вычитка

**Исполнитель:** Рецензент

**Действия:**

- Чтение результата 1-го прогона через `python-docx`
- Проверка по всем пунктам изначального списка нарушений:
  - программная проверка полей, шрифтов, интервалов
  - поиск кириллицы вне `[]` в References
  - подсчёт слов в аннотациях
  - проверка нумерации ссылок
  - поиск дублирований текста (признак «добавил, но не удалил»)
- Формирование списка непройденных пунктов

**Результат:** точечный промпт-3 для 2-го прогона исполнителя.

---

### Этап 9. Точечные правки (2-й прогон, при необходимости)

**Исполнитель:** Технический исполнитель

**Действия:**

- Получение промпта-3 с конкретными правками (обычно 4–8 штук)
- Точечное внесение правок
- Сохранение финального `.docx`

**Ожидаемый результат:** 100% закрытие пунктов чек-листа.

---

### Этап 10. Финальная проверка

**Исполнитель:** Рецензент

**Действия:**

- Повторная программная вычитка финального документа
- Подтверждение, что все N нарушений закрыты
- При необходимости — ещё один цикл (обычно не требуется)

## 4. Ключевые принципы конвейера

### Принцип 1: «Подбирать модель по сильной стороне»

- **Аналитика + чтение файлов + проверка форматирования** → модель с сильным tool use
- **Генерация длинных текстов с соблюдением лимитов** → модель с хорошим следованием инструкциям
- **Механические операции над `.docx`** → специализированный инструмент или модель с доступом к `python-docx` / OOXML

### Принцип 2: «Большая модель не обязательно лучше»

- Для задач с жёсткими требованиями к формату (ГОСТ, лимиты слов, точная транслитерация) меньшие и более «послушные» модели часто превосходят большие
- Большие модели склонны «креативничать»: переписывают числа прописью, вольно переводят названия, меняют стиль на философский

### Принцип 3: «Человек-в-цикле необходим»

- Решение о снятии пунктов из чек-листа принимает только человек
- Выбор следующей модели/инструмента для тестирования определяется совместно, но финальное решение — за человеком
- Контрольная вычитка выявляет пропуски, которые ни одна модель не заметила бы сама

### Принцип 4: «Встраивай критические данные в промпт»

- Sub-agents (`researcher-web`, `researcher-docs`) полезны для проверки, но не для генерации основных данных
- Библиография, ГОСТ-правила, требования к оформлению — всё это должно быть явно в промпте, а не «найдено» sub-agent'ом
- Sub-agent, ищущий «более подходящие» источники, может заменить авторов и заголовки (критическая галлюцинация)

### Принцип 5: «Не доверяй самоотчёту модели»

- Каждая модель имеет свои слепые зоны
- Библиография должна проверяться на соответствие оригиналу, а не только на формат
- Code-reviewer проверяет формат, но пропустит подмену авторов, если явно не поручить ему эту проверку
- Самоотчёт «я сделал X» без программной проверки — **недостоверен**

### Принцип 6: «Итеративность — норма»

- 1-й прогон исполнителя редко даёт 100% — обычно 80–90%
- Точечные правки во 2-м прогоне закрывают остаток
- Не пытаться добиться 100% за один прогон — это дороже, чем два прохода

### Принцип 7: «Двойная верификация факт-чекера (Zero Trust для текстовика)»

- **Этап 3 (Pre-generation):** Факт-чекер сначала парсит ссылки и материалы из исходника, формируя `results.md` (сырой контент). Текстовик собирает `[TEXT-N]` строго из этого валидированного сырья.
- **Этап 4.5 (Post-generation):** Сгенерированные `[TEXT-N]` верифицируются факт-чекером второй раз, чтобы убедиться, что Текстовик не добавил новых галлюцинаций.
- Факт-чекер может быть вызван **точечно** в любой точке конвейера по решению Оркестратора + Рецензента
- Возвращает вердикт с прямыми цитатами и URL — без выдумываний и ложных отрицаний
- **Эмпирическое обоснование:** в тесте на 27 источников факт-чекер обнаружил 6 галлюцинированных статей (22%), которые прошли все предыдущие проверки. Без этого гейта документ не прошёл бы рецензирование журнала

## 5. Что делать после этого документа

Ты ознакомлен с архитектурой конвейера. Сейчас ожидай конкретное задание в формате:

```
Ты — [РОЛЬ]. Задача: ... Доп. уточнения: ...
```

где **РОЛЬ** — одна из:

- `ТЕКСТОВИК`
- `РЕЦЕНЗЕНТ`
- `СО-ОРКЕСТРАТОР`
- `ТЕХНИЧЕСКИЙ ИСПОЛНИТЕЛЬ`

### Возможные форматы задания

| Формат | Шаблон | Что делать |
|--------|--------|------------|
| **A** | `Ты — ТЕКСТОВИК. Задача: ...` | Генерируй `[TEXT-N]` блоки по ТЗ |
| **B** | `Ты — РЕЦЕНЗЕНТ. Задача: ...` | Анализируй документ / оценивай выходы другой модели |
| **C** | `Ты — СО-ОРКЕСТРАТОР. Задача: ...` | Помогай декомпозировать задачу и готовить промпты |
| **D** | `Ты — ТЕХНИЧЕСКИЙ ИСПОЛНИТЕЛЬ. Задача: ...` | Вноси механические правки в `.docx` по чек-листу |

---

>
> Если всё ясно — **подтверди готовность** коротким ответом и ожидай конкретную задачу.
